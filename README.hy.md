# GLDtk — Generative Level Design Toolkit
### Հայերեն նկարագրություն

---

## Ինչ է GLDtk-ն

GLDtk-ն գործիք է, որը **տեքստային հրամանից ստեղծում է խաղային մակարդակ** (level)՝ LDtk ֆորմատով։ Գրում ես՝ «Մութ քարանձավ, փշոտ փոսեր, ելքը վերևում է» — համակարգը ստեղծում է ամբողջական, ֆիզիկապես ճիշտ, խաղալու պատրաստ 2D platformer մակարդակ։

Կախ չէ նրանից, թե ինչ LLM ես օգտագործում — Claude, GPT, DeepSeek, կամ տեղական Ollama — արդյունքը **միշտ ֆիզիկապես հնարավոր** կլինի։

---

## Գաղափարի էությունը

Ժամանակակից AI-ի հիմնախնդիրն ստեղծագործական խնդիրների լուծման ժամանակ այն է, որ **նյարդային ցանցերը (neural networks) ստում են**։ LLM-ը կարող է ստեղծել մի հրաշք գաղափար, բայց ասել, որ կերպարը կարող է 10 մետր թռիչք անել — ինչը ֆիզիկապես անհնար է։

GLDtk-ն լուծում է սա **neuro-symbolic** մոտեցմամբ.

```
LLM (neural)          →  ստեղծագործականություն, բազմազանություն
Ֆիզիկայի օրակլ (symbolic) →  ճշտություն, հնարավորություն, դետերմինիզմ
```

LLM-ն ստեղծում է մակարդակի **վերացական գրաֆ** — հարթակների և թռիչքների ցանց։ Ֆիզիկայի օրակլը ստուգում է **յուրաքանչյուր** կապ. հնարավո՞ր է այս թռիչքը ֆիզիկայի օրենքների համաձայն։ Եթե ոչ — կամ ինքնաբար ուղղում է, կամ LLM-ին ուղղակի սխալի հաղորդագրություն է ուղարկում կրկին փորձելու համար։

**Արդյունք — և ստեղծագործական, և երաշխավորված խաղալու հնարավոր մակարդակ։**

## Ֆիզիկայի մաթեմատիկան

### Թռիչքի հնարավորության ստուգում

Ամեն `Jump` կապ ստուգվում է կինեմատիկ հավասարմամբ։ Հարթակից A-ից B հնարավո՞ր է հասնել.

```
½g·t² − v_jump·t + Δy = 0
```

Որտեղ.
- `g` — ձգողականության արագացում (980 px/s²)
- `v_jump` — ցատկի սկզբնական ուղղահայաց արագություն (600 px/s)
- `Δy` — բարձրության տարբերություն (B.y − A.y)
- `t` — ժամանակ

Հավասարման **դիսկրիմինանտը**.

```
D = v_jump² − 2·g·Δy
```

Եթե `D < 0` — ցատկի ագույցը (apex) թիրախից ցածր է, թռիչքը **ֆիզիկապես անհնար է**։ Հաշվարկն ակնթարթային է (millisecond) և **100% դետերմինիստ** — ոչ հավանականություն, ոչ «գուցե»։

### Ընկնելու ժամանակ

```
Δy = ½·g·t²  →  t = √(2·Δy / g)
```

Օգտագործվում է `Fall` կապերի ստուգման համար։

### Սկսնակ ուղղահայաց արագություն

```
v = √(2·g·Δy)
```

Հաշվում է, թե ինչ արագությամբ պետք է ցատկել կոնկրետ բարձրությանը հասնելու համար։

## Sugiyama Layout Ալգորիթմ

Երբ LLM-ը ստեղծում է գրաֆ (հարթակների ցանց), պետք է դրանք դասավորել tile grid-ում։ Դրա համար օգտագործվում է **Sugiyama 4-փուլ ալգորիթմ**:

```
Փուլ 1 — Ցիկլերի հեռացում
  Գրաֆը կարող է պարունակել ցիկլեր (A→B→A)
  DFS-ով հայտնաբերում ենք հետ-կապերը (back edges) և ժամանակաւorabi հեռացնում

Փուլ 2 — Շերտերի (Layer) վերագրում
  Ամեն հանգույցի (platform) տրվում է «խորություն» Start-ից
  Golden path (Start→Exit) = մակարդակի ողնաշար

Փուլ 3 — Հատումների նվազեցում (Barycenter)
  Յուրաքանչյուր շերտում հանգույցները դասավορվում են
  Barycenter մեթոդով (հարևան հանգույցների կշռված միջին)

Փուլ 4 — Կոորդինատների վերագրում
  (layer, order) → (col, row) tile grid-ում
  JUMP կապ → row-ը ցած է (ֆիզիկապես վեր)
  FALL կապ → row-ը վեր է (ֆիզիկապես վար)
```

**Golden Path** (Start-ից Exit ամենակարճ ճանապարհը) **միշտ** layout-ի կենտրոնական ողնաշարն է — ապահովում է, որ մայն ճանապարհը տրամաբանական ու հետևողական լինի։

## Autotile համակարգ

Tile grid-ից visual tiles ստեղծելու համար օգտագործվում է **4-bit NSEW bitmask**:

```
N = 8   (վերևի հարևան)
S = 4   (ներքևի հարևան)
E = 2   (աջ հարևան)
W = 1   (ձախ հարևան)
```

16 հնարավոր կոմբինացիա → 14 tile role (CORNER_TL, EDGE_T, SINGLE, FILL, HAZARD_TILE, և այլն)։ Ավտոмат կերպով ճիշտ tile-ն ընտրվում է կախված հարևաններից — ոչ ձեռով, ոչ կանոններ գրելով։

## Entity-ների (կերպարների) դասավորություն

Coins, enemies, checkpoints դասավorum են **Golden Path-ի դժվարության կշռով**.

```python
difficulty = base_position + edge_bonus

# edge_bonus:
# JUMP կապ → +0.15 (ավելի դժվար)
# FALL կապ → +0.05

# Coin   → difficulty < 0.9  (ամենուր)
# Key    → ամենամոտը 0.40-ին
# Enemy  → difficulty > 0.35
# Checkpoint → ամենամոտը 0.50-ին (եթե ≥5 հանգույց կա)
```

## Ինչպես է աշխատում ամբողջ համակարգը

```
Տեքստ ("Մութ քարանձավ...")
      │
      ▼
LLMController — LLM-ն ստեղծում է JSON գրաֆ
      │ Ստուգում. կառուցվածք + ֆիզիկա
      │ Եթե սխալ → SymbolicPatcher (ինքնաբար ուղղում)
      │ Եթե չի ուղղվում → LLM-ին կրկին ուղարկում
      │
      ▼
AbstractLevelGraph — ֆիզիկապես վավեր գրաֆ
      │
      ▼
Sugiyama Layout — tile grid-ի կոորդինատներ
      │
      ▼
Aesthetic Layer — theme, autotile, entities
      │
      ▼
LDtk Adapter — LDtk JSON v1.5.3
      │
      ▼
level.ldtk — բացիր LDtk editor-ում, խաղա
```

## LDtk Ինտեգրում

Ծրագիրը ինտեգրված է **LDtk editor-ի մեջ** որպես native Haxe panel — ոչ browser extension, ոչ արտաքին tool։

- LDtk-ի աջ կողմում կա **«AI»** կոճակ
- Բացվում է GLDtk panel — theme, difficulty, enemies, size, prompt
- Սեղմում ես Generate → Python backend-ն ստեղծում է մակարդակ → LDtk-ն ռեload է անում
- Ամենն ստեղծվում է ուղղակի բացված project ֆայլում

## Ապագա Ծրագրեր

### 1. Ֆինֆ-Թյուն (Fine-tuning)

**Նպատակ** — փոքր, տեղական LLM-ը դարձնել հուսալի level generator, որ ամեն անգամ ճիշտ գրաֆ ստեղծի առաջին փորձից։

```
Քայլ 1: Հավաքել tile grid-ներ (LDtk, Tiled, ձեռագործ)
Քայլ 2: extract_graph() → AbstractLevelGraph
Քայլ 3: Ֆիլտրել — թողնել միայն is_solvable() + oracle-ն ընդունողները
Քայլ 4: Fine-tune LoRA-ով Llama / Qwen / Mistral-ի վրա
Քայլ 5: Ollama-ի միջոցով տեղադրել և օգտագործել offline
```

### 2. Ամրապնդող ուսուցում (Reinforcement Learning)

Ֆիզիկայի oracle-ն **безkfree reward function է** — մարդու annotation-ի կարիք չկա։

```
State:  կիսատ գրաֆ
Action: ավելացնել/տեղափոխել/հեռացնել հանգույց կամ կապ
Reward: +1 եթե is_solvable() AND բոլոր edges-ը oracle-ն անցնում
        −0.1 ամեն անհնար edge-ի համար
```

### 3. Distillation (Մեծ → Փոքր)

```
1. Claude-ով ստեղծել հարյուրավոր վավեր (prompt, graph) զույգ
2. Fine-tune 7B մոդել այս տվյալների վրա
3. LLMController(provider="ollama", model="gldtk-7b")
4. Symbolic patcher-ը մնում է safety net-ով
```

### 4. Մակարդակի խմբագրում (Selection-based editing)

Ընտրել LDtk-ում հատված մակարդակ → GLDtk panel-ին ասել «փոխիր միայն սա» → pipeline-ը regenerate է անում միայն ընտրված հատվածը — մնացածն անձեռնմխ։

### 5. Ավելի շատ theme-ներ ու tileset-ներ

Ներկայիս placeholder tileset-ները (dungeon, forest, sky) կփոխարինվեն իրական pixel art-ով։ Tile layout-ի convention-ը (5×3 region, base_x offset) մնում է — code-ը կփոխվի ոչ, միայն PNG-ն կփոխվի։

### 6. Ավելի 똑똑한 layout

Sugiyama-ն linear layout է ստեղծում։ Ապագայում — branch-avar ճանապարհներ, secret անցումներ, non-linear կառուցվածք, multi-room levels։
